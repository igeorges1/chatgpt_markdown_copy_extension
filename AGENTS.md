# Repository Guidelines

## Project Structure & Module Organization
- `extension/`: core extension source (content/background/options/popup scripts, HTML/CSS, i18n, icons).
- `tests/`: unit-style Node test runner cases (`*.test.js`) for markdown formatting and i18n behavior.
- `e2e/`: end-to-end tests (`e2e.test.js`) using Playwright-style flows.
- `fixtures/`: saved ChatGPT/Gemini HTML and parsed markdown outputs for deterministic test inputs.
- `dist/`: build output generated by `pnpm build` (do not hand-edit).
- `scripts/`: utility scripts for fixture parsing and token refresh workflows.

## Build, Test, and Development Commands
- `pnpm lint`: run ESLint on `extension/**/*.js`.
- `pnpm lint:fix`: apply safe lint autofixes.
- `pnpm test`: run unit tests via Node’s built-in test runner.
- `pnpm test:e2e`: run e2e tests in `e2e/`.
- `pnpm build`: bundle extension scripts with `tsup`, copy manifest/assets/locales, and normalize output filenames in `dist/`.
- `pnpm lint:web-ext`: run `web-ext` validation for extension packaging.

## Coding Style & Naming Conventions
- Language: JavaScript (ES modules, `"type": "module"`).
- Indentation: 2 spaces; keep semicolon and quote style consistent with surrounding files.
- Filenames: lowercase, descriptive; tests end with `.test.js`.
- Keep browser-extension responsibilities separated by entrypoint (`content.js`, `background.js`, `options.js`, `popup.js`).
- Run `pnpm lint` before opening a PR.

## Testing Guidelines
- Framework: Node test runner (`node --test`) with `jsdom` for DOM-focused behavior.
- Add/update fixtures in `fixtures/` when parser behavior changes.
- Name tests by behavior, e.g. `markdown-spacing.test.js`, and keep assertions deterministic.
- Minimum expectation for changes: `pnpm test` passes; run `pnpm test:e2e` for UI/flow-impacting changes.

## Commit & Pull Request Guidelines
- Follow existing history style: short, imperative, sentence-case summaries (e.g., `Add workflow to verify secrets have actual values`).
- Keep commits scoped to one logical change.
- PRs should include:
  - purpose and user-visible impact,
  - test evidence (`pnpm test`, plus `pnpm test:e2e` when relevant),
  - screenshots/GIFs for popup/options UI changes,
  - linked issue or workflow context when applicable.

## Security & Configuration Tips
- Never commit live secrets/tokens. Use `sentry-config.example.js` as a template.
- Validate extension packaging with `pnpm lint:web-ext` before release-related changes.
