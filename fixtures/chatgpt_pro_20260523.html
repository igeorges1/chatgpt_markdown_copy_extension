你说：

什么能力/习惯/决策会决定了谁用coding agent能实现最好的效果？

ChatGPT 说：

我会从个人能力、工作习惯、关键决策/组织环境三层来拆解，并结合主流 coding agent 的实践建议，提炼出可操作的判断框架。

最能把 coding agent 用出效果的人，不一定是写代码最快的人，而是能把模糊需求变成清晰、可验证、低风险任务的人。更准确地说：coding agent 放大的是你的“软件工程判断力”，不是单纯放大打字速度。

我会分三层看：能力、习惯、决策。

1. 最核心能力：把任务定义清楚

高水平用户给 agent 的不是“帮我改一下这个功能”，而是：

目标是什么？
相关上下文在哪里？
有哪些约束？
做到什么程度算完成？

这和 OpenAI Codex 官方建议非常接近：一个好任务说明通常包括 goal、context、constraints、done when；这样可以减少 agent 的假设，让结果更容易审查。OpenAI 开发者

差距通常体现在这里：

普通用法：

帮我优化登录逻辑。

高手用法：

目标：修复用户登录后偶发跳回登录页的问题。
上下文：auth/session.ts、middleware.ts、最近的错误日志如下。
约束：不要改动 OAuth provider 配置；保持现有 cookie 名称；不要引入新依赖。
完成标准：新增回归测试；本地 pnpm test auth 通过；解释 bug 根因和改动风险。

决定效果的不是 prompt 长，而是任务边界、上下文和验收标准是否足够清楚。

2. 第二能力：知道什么该交给 agent，什么不该交

coding agent 擅长的是边界清楚、可测试、可回滚、上下文主要在 repo 内的任务，例如修 bug、补测试、重构局部模块、更新文档、加小功能、迁移重复样板代码。GitHub 对 Copilot coding agent 的描述也强调它可以研究仓库、制定实现计划、改分支代码，并让人类审查 diff、迭代后再开 PR。GitHub Docs

效果差的人常犯的错误是把下面这些直接丢给 agent：

任务类型 为什么风险高
“重构整个支付系统” 范围太大，业务语义和隐性约束多
“帮我设计核心架构” 需要长期取舍，不只是代码生成
“修一下线上事故” 时间、风险、安全性都高
“把这个功能做完整”但没有验收标准 agent 会自行补假设
涉及权限、认证、PII、安全边界 错误代价高

GitHub 的最佳实践也明确把复杂宽泛任务、需要深度业务知识的任务、敏感/关键任务、定义不清的任务列为更适合人类亲自处理或强监督的类别。GitHub Docs

所以高水平决策是：不是问“agent 能不能做”，而是问“这个任务能不能被我切成 agent 可安全完成的小块”。

3. 第三能力：会让 agent 先研究和计划，而不是立刻开写

很多人用 coding agent 效果不好，是因为一上来就让它“改代码”。高手通常会先要求：

先阅读相关文件，说明现有实现。
找出可能改动点。
给出 2–3 个方案。
先不要写代码。
等我确认方案后再实现。

OpenAI Codex 的官方建议是：复杂、模糊、难描述的任务，先让 Codex 计划；GitHub 也建议在开 PR 之前先让 agent 研究 repo、制定实现计划、迭代代码。OpenAI 开发者+1

这背后的关键是：
agent 很擅长执行，但它不总是擅长替你判断“该不该这么做”。

你越早让它进入执行，它越容易沿着错误方向跑很远。
你先让它计划，就能在最便宜的时候纠偏。

4. 第四能力：验证能力，而不是信任能力

用 coding agent 最好的人，往往有很强的 review 能力。他们不把 agent 的输出当答案，而是当“一个候选 patch”。

他们会检查：

检查项 具体问题
行为是否正确 真的修了用户问题吗？
测试是否覆盖 是测实现细节，还是测业务行为？
是否引入回归 旧路径、边界条件、错误处理有没有坏？
diff 是否过大 有没有顺手改了不该改的东西？
安全性 是否泄露 secret、放宽权限、绕过校验？
可维护性 是否产生了奇怪抽象或重复逻辑？

OpenAI 的 Codex 文档也把写/更新测试、运行测试、检查 lint/format/type check、确认行为符合请求、审查 diff 列为重要验证步骤。OpenAI 开发者

一句话：
不会 review 的人，用 agent 是在赌；会 review 的人，用 agent 是在加速。

5. 第五习惯：把重复的上下文沉淀成仓库规则

高手不会每次都重复告诉 agent：

我们用 pnpm。
API 放在 src/server。
不要改 generated files。
测试命令是 pnpm test:unit。
PR 需要包含风险说明。
React 组件命名规则是……

他们会把这些写进 repo 级说明，例如 AGENTS.md、custom instructions、项目 README、测试说明、架构说明等。OpenAI 建议用 AGENTS.md 记录 repo layout、运行方式、build/test/lint 命令、工程约定、PR 预期、约束和验收方式；GitHub 也建议通过 repository custom instructions 帮助 agent 理解项目、构建、测试和验证改动。OpenAI 开发者+1

这件事的本质是：

不要只优化 prompt，要优化 agent 的工作环境。

一个 agent-friendly repo 通常有：

资产 作用
清晰 README 让 agent 快速知道项目怎么跑
测试命令 让 agent 能自证改动
lint/typecheck 提供自动反馈
架构说明 减少错误抽象
API/example fixtures 减少猜测
PR checklist 让输出更符合团队标准
禁止事项 防止 agent 改错边界

6. 第六习惯：小步快跑，而不是一次性交付

效果最好的 workflow 通常不是：

给 agent 一个大任务 → 等它全做完 → 看结果。

而是：

让它研究 → 给计划 → 确认方案 → 实现最小版本 → 跑测试 → review diff → 评论修改 → 再迭代。

GitHub 也把“researching, planning, iterating before opening a pull request”作为重要流程，并建议像对待人类开发者一样，通过 PR 评论继续迭代 agent 的工作。GitHub Docs

真正的高手会把 agent 当成一个速度很快但需要管理的 junior/mid-level 工程师，而不是一个魔法按钮。

7. 第七能力：上下文管理

coding agent 的很多失败不是“模型不会写代码”，而是它拿到的上下文错了、不全、过期、太噪。

高水平用户会主动提供：

上下文类型 示例
相关文件 “重点看 billing/usage.ts 和 plans.ts”
错误日志 stack trace、test failure、runtime logs
业务规则 “free plan 不能创建超过 3 个项目”
历史决策 “这里不用 Redis 是因为多区域一致性问题”
反例 “不要采用之前 PR #123 的做法，因为……”
验收方式 “curl 这个 endpoint 应返回 403”

Anthropic 在 agent context engineering 文章里也强调，复杂任务需要保留架构决策、未解决 bug、实现细节等关键上下文；结构化笔记可以帮助 agent 在长任务中追踪进度和依赖。Anthropic

这就是为什么很多高手会让 agent 维护：

PLAN.md
NOTES.md
TODO.md
“当前假设列表”
“已验证/未验证事项”

不是为了仪式感，而是为了减少长任务中的上下文漂移。

8. 第八能力：有工程品味，能识别“看起来对但其实烂”的代码

coding agent 很容易生成“局部正确、整体变差”的代码，比如：

表面现象 深层问题
测试通过 但 mock 掉了真正风险
功能能跑 但破坏了领域模型
代码很完整 但抽象过度
diff 很大 但核心 bug 只需改一行
类型没报错 但业务不变量被破坏
加了 fallback 但掩盖了真正错误

所以 coding agent 时代，高价值能力会从“亲手写每一行”转向：

判断什么是好代码。
判断什么是好设计。
判断什么风险不能接受。
判断什么时候应该删掉 agent 写的东西。

这也是为什么资深工程师往往比初学者更容易从 agent 获益：他们更知道哪里可能错。

9. 第九决策：为 agent 建立安全边界

好的使用者会提前决定：

边界 例子
权限边界 agent 不能接触生产 secret
分支边界 只允许在 feature branch 上改
文件边界 不允许改 migration、generated files、auth core
运行边界 不自动执行破坏性命令
发布边界 agent 不能直接 merge/deploy
审查边界 所有 agent PR 必须人工 review

GitHub cloud agent 的产品设计本身也体现了这种思路：它可以在分支上研究、计划、改代码，人类再审查 diff、迭代并决定是否开 PR。GitHub Docs

越是强大的 agent，越需要清晰的 guardrails。
没有边界的自动化，放大的不是效率，而是事故半径。

10. 最关键的组织决策：让代码库变得“可被 agent 理解”

个人 prompt 技巧有用，但组织层面的收益更大。最强团队会系统性改善：

决策 对 agent 的影响
增加测试覆盖 agent 可以自我验证
统一代码风格 agent 少猜测
写清架构文档 agent 少走弯路
改善 CI 速度 迭代周期变短
提供本地启动脚本 agent 能运行项目
维护 fixtures/examples agent 能模仿正确模式
把业务规则写进代码/文档 减少隐性知识
拆小 issue 提高成功率
建立 AI PR review 规范 降低合并风险

所以长期来看，决定谁用得最好的不是“谁会写神奇 prompt”，而是：

谁的工程系统更清晰、更可测试、更可审查、更模块化。

一个简单判断公式

我会用这个公式判断某人/某团队能不能用好 coding agent：

效果 = 任务清晰度 × 上下文质量 × 验证能力 × 迭代速度 × 风险边界

任何一项接近 0，整体效果都会很差。

比如：

类型 结果
任务清楚，但没测试 agent 写得快，但你不敢合
有测试，但需求模糊 agent 会快速实现错东西
上下文充分，但没人 review 容易合入隐蔽 bug
review 很强，但 repo 难跑 迭代成本高
工程系统好，但任务太大 agent 容易迷路

最终结论

最会用 coding agent 的人，通常具备这些特征：

  1. 会拆任务：把大问题切成小、清楚、可验收的任务。

  2. 会给上下文：告诉 agent 看哪里、不要看哪里、遵守什么约束。

  3. 会先计划：复杂任务先让 agent 研究和提方案，而不是马上写代码。

  4. 会验证：看 diff、跑测试、查边界条件，不盲信输出。

  5. 会沉淀规则：把重复指令写进 repo instructions、AGENTS.md、README、CI。

  6. 会选择任务：把低风险、高重复、可测试任务交出去，把高风险决策留给人。

  7. 有工程品味：能识别“能跑但不该合”的代码。

  8. 有安全边界：不让 agent 直接碰生产、密钥、核心权限、不可逆操作。

  9. 会迭代协作：像管理一个很快的工程师一样管理 agent。

  10. 愿意改造工作流：不是把 agent 塞进旧流程，而是让代码库、issue、CI、review 流程都更适合 agent。

一句话概括：

coding agent 时代，最重要的能力不是“写代码”，而是“定义正确问题、提供正确上下文、验证正确结果”。
